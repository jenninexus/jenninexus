Initial diagnosis
-----------------
- There are multiple identity files for the same host (64.23.141.41). Without a per-host IdentityFile override the SSH client will try multiple keys in sequence. If the server has a different public key recorded (or limited number of allowed key attempts), you'll see authentication failure.
- Your `~/.ssh/config` already contains host aliases which is good — but some aliases point to `User root` while deploy workflows expect `User deploy`. Mixing root vs deploy and multiple identities is the likely source of confusion.

Goals / success criteria
------------------------
1. `ssh root@64.23.141.41` (or `ssh main-jenninexus`) connects reliably using the intended key for root.
2. `ssh deploy@64.23.141.41` (or `ssh main-jenninexus-ephemeral`) connects reliably for deploy uploads.
3. WinSCP/VSCode use the same per-host config so they don't prompt or upload with the wrong identity.

Recommended per-host configuration (local)
-----------------------------------------
Add (or update) these blocks in `C:\Users\Owner\.ssh\config` so each workspace/project has an unambiguous alias and IdentityFile.

Example entries (append to the end of `~/.ssh/config`):

Host main-jenninexus-root
	HostName 64.23.141.41
	User root
	IdentityFile C:/Users/Owner/.ssh/main_jenninexus
	IdentitiesOnly yes

Host main-jenninexus-deploy
	HostName 64.23.141.41
	User deploy
	IdentityFile C:/Users/Owner/Projects/www/jenninexus/.deploy_keys/main_jenninexus_ephemeral
	IdentitiesOnly yes
	StrictHostKeyChecking accept-new

Host main-boobsofwar-deploy
	HostName 64.23.141.41
	User deploy
	IdentityFile C:/Users/Owner/.ssh/main_boobsofwar
	IdentitiesOnly yes

Notes:
- Use the `*-root` alias when you need root access; `*-deploy` for the deploy user (safer for uploads).
- `IdentitiesOnly yes` prevents the client from offering keys outside the specified IdentityFile(s).

How to test which key the client offers (do this from PowerShell)
----------------------------------------------------------------
1) Verbose SSH connection (shows offered identities and server response):

	ssh -vvv main-jenninexus-root

2) Test using explicit identity file (fast):

	ssh -i C:/Users/Owner/.ssh/main_jenninexus root@64.23.141.41

3) Test the deploy alias:

	ssh main-jenninexus-deploy

Interpretation:
- Look for lines like "Offering public key: C:/Users/.../main_jenninexus" and server responses. If server replies "Authentications that can continue: publickey" or "Permission denied (publickey)", the server didn't accept the offered key.

If the server is missing the public key (what to do on the server)
------------------------------------------------------------
If you have another administrative route (console or control panel) use that. Otherwise ask an admin to run the following steps on the server as root (or run them yourself if you can login as root via some other method):

# For root access (add the public key for root)
sudo mkdir -p /root/.ssh
sudo sh -c 'echo "<PASTE-ROOT-PUBKEY-HERE>" >> /root/.ssh/authorized_keys'
sudo chown -R root:root /root/.ssh
sudo chmod 700 /root/.ssh
sudo chmod 600 /root/.ssh/authorized_keys

# For deploy user (safe deploy access)
sudo mkdir -p /home/deploy/.ssh
sudo sh -c 'echo "<PASTE-DEPLOY-PUBKEY-HERE>" >> /home/deploy/.ssh/authorized_keys'
sudo chown -R deploy:deploy /home/deploy/.ssh
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys

Replace `<PASTE-*-PUBKEY-HERE>` with the exact contents of the `.pub` file you intend to authorize (for example `main_jenninexus.pub` for root or `main_jenninexus_ephemeral.pub` for deploy).

WinSCP / Pageant notes
----------------------
- If you use WinSCP and it requires a PPK, convert the OpenSSH private key to PPK using PuTTYgen or use the native OpenSSH key support in the latest WinSCP.
- Make sure WinSCP is configured to use the matching host alias from your `~/.ssh/config` or explicitly set the private key path in the WinSCP site entry.
