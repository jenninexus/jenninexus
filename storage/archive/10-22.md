Boobsofwar (live) — immediate investigation checklist

We observed the `boobsofwar` site returning 404. The nginx config for `boobsofwar` lives alongside the `jenninexus` nginx config on the server. Below are bullet-proof steps to find, compare, and fix the issue safely.

1) Find the nginx site configuration files (server)

  # list site configs and find the boobsofwar/jenninexus files
  ls -l /etc/nginx/sites-enabled /etc/nginx/sites-available
  grep -R "boobsofwar" /etc/nginx -n || true
  grep -R "jenninexus" /etc/nginx -n || true

2) Inspect and compare the two site configs

  # view the configs (replace the filenames if different)
  sudo sed -n '1,240p' /etc/nginx/sites-available/boobsofwar.conf
  sudo sed -n '1,240p' /etc/nginx/sites-available/jenninexus.conf

  Things to compare (quick checklist):
  - server_name values and any duplicate server blocks
  - root directive paths (must point to the correct document root, e.g. /var/www/boobsofwar/public_html)
  - index directive (index.php,index.html) and try_files rules
  - location blocks that may rewrite/deny requests (try_files vs return 404)
  - fastcgi / php-fpm upstream settings (if PHP is used) and socket vs tcp

3) Inspect the site document root and permissions

  # verify document root files and ownership (on server)
  stat -c '%A %U %G %a %n' /var/www/boobsofwar /var/www/boobsofwar/public_html || true
  ls -la /var/www/boobsofwar/public_html | sed -n '1,240p'

  Expected:
  - root should be readable by `www-data` (owner or at least group read+execute)
  - index.php or index.html must exist per nginx `index` directive

4) Reproduce the request and examine nginx logs

  # test nginx config and check error log
  sudo nginx -t
  sudo tail -n 200 /var/log/nginx/error.log | sed -n '1,240p'
  sudo tail -n 200 /var/log/nginx/access.log | sed -n '1,240p'

  Common 404 causes found in logs:
  - root points to wrong directory
  - try_files rewrites send requests to upstream that returns 404
  - index file missing or permissions prevent read

5) Apply safe fixes

  - If `root` is wrong, update the site config, run `sudo nginx -t` and `sudo systemctl reload nginx`.
  - If permissions are wrong, `chown -R www-data:www-data /var/www/boobsofwar` (or appropriate owner) and `find /var/www/boobsofwar -type d -exec chmod 755 {} \; && find /var/www/boobsofwar -type f -exec chmod 644 {} \;` then test again.
  - If PHP upstream is misconfigured, compare jenninexus's `fastcgi_pass` and socket path and use the same working pattern.

6) If the site still 404s after the above

  - Temporarily enable a minimal server block pointing to a simple index.html to confirm nginx serves static content from that path.
    * create /var/www/boobsofwar/public_html/debug-index.html with simple text, update root if needed and test.
  - If debug page serves but app still 404s, the issue is in app routing or rewrites.

7) Record changes and roll back plan

  - Always keep a timestamped backup of any config before editing: `sudo cp /etc/nginx/sites-available/boobsofwar.conf /root/boobsofwar.conf.bak.$(date -u +%Y%m%dT%H%M%SZ)`
  - After changes: `sudo nginx -t && sudo systemctl reload nginx` and verify logs.
